## 详细对比表

### 消息处理流程

| 阶段 | nanobot (当前) | OpenClaw (参考) | 改进后 |
|------|---------------|----------------|--------|
| **接收** | WebSocket 直接回调 | Event Dispatcher 统一分发 | ✅ Event Dispatcher |
| **去重** | 内存 OrderedDict | 内存 + SQLite 双保险 | ✅ 内存 + SQLite |
| **防抖** | ❌ 无 | Debouncer 合并连续消息 | ✅ 300ms 防抖 |
| **队列** | ❌ 无（并发处理） | Per-Chat 串行队列 | ✅ 每 chat 独立队列 |
| **会话** | 简单 chat_id | 4 种隔离策略 | ✅ 可配置策略 |
| **权限** | 简单白名单 | Pairing + 多级策略 | ✅ 完整权限系统 |

### 可靠性

| 特性 | nanobot | OpenClaw | 改进后 |
|------|---------|----------|--------|

| 断线重连 | while + sleep | SDK + 指数退避 | ✅ 指数退避 |
| 并发控制 | ❌ | ✅ 队列 | ✅ 队列 |
| 死信处理 | ❌ | ✅ 持久化 | ✅ 持久化 |

## 性能对比

### 消息处理延迟

```
场景: 用户快速发送 5 条消息

nanobot (当前):
  msg1 ────────────────────────▶ 处理 ─▶ 回复
  msg2 ────────────────────────▶ 处理 ─▶ 回复  (并发，可能乱序)
  msg3 ────────────────────────▶ 处理 ─▶ 回复
  msg4 ────────────────────────▶ 处理 ─▶ 回复
  msg5 ────────────────────────▶ 处理 ─▶ 回复
  
  问题: Agent 被调用 5 次，可能回复顺序混乱

改进后:
  msg1 ─┐
  msg2 ─┼──▶ [Debounce 300ms] ──▶ 合并 ──▶ [Queue] ──▶ 处理 ─▶ 回复
  msg3 ─┘                                    (串行，顺序保证)
  msg4 ────────────────────────────────────▶ 处理 ─▶ 回复
  msg5 ────────────────────────────────────▶ 处理 ─▶ 回复
  
  优势: 前 3 条合并为 1 次调用，节省 Token，保证顺序
```

## 关键改进点详解

### 1. 去重：内存 → 持久化

**Before:**
```python
self._processed_message_ids: OrderedDict[str, None] = OrderedDict()
# 服务重启 → 全部丢失
```

**After:**
```python
self._dedup = SQLiteDeduplicationStore("~/.nanobot/dedup.db")
# 服务重启 → 从 SQLite 恢复
# 支持 TTL 自动过期
```

**收益:**
- ✅ 重启不丢消息状态
- ✅ 支持多实例共享
- ✅ 可审计历史记录

---

### 2. 处理：并发 → 队列保序

**Before:**
```python
def _on_message_sync(self, data):
    # 直接调度，无顺序保证
    asyncio.run_coroutine_threadsafe(self._on_message(data), self._loop)
```

**After:**
```python
async def _on_message(self, data):
    chat_id = data["message"]["chat_id"]
    # 进入队列，同一 chat 串行处理
    await self._queue.enqueue(chat_id, lambda: self._handle(data))
```

**收益:**
- ✅ 消息严格按接收顺序处理
- ✅ 避免竞态条件
- ✅ 支持背压控制

---

### 3. 会话：简单 → 策略化

**Before:**
```python
reply_to = chat_id if chat_type == "group" else sender_id
# 仅 2 种情况
```

**After:**
```pythonnresolver = FeishuSessionResolver(scope=SessionScope.GROUP_TOPIC)
session_key = resolver.resolve(
    chat_id, sender_id, root_id, thread_id
)
# 4 种策略，灵活配置
```

**收益:**
- ✅ 群内多用户独立会话
- ✅ 话题级别上下文隔离
- ✅ 适应不同使用场景

---

### 4. 权限：白名单 → 完整策略

**Before:**
```python
def _is_allowed(self, sender_id: str) -> bool:
    return sender_id in self.config.allow_from
```

**After:**
```pythonnchecker = PolicyChecker(config, pairing_manager)
allowed, code = checker.check_dm(user_id)
if not allowed and code:
    await send_pairing_request(user_id, code)
```

**收益:**
- ✅ 新用户配对流程
- ✅ 群组级别权限
- ✅ 发送者白名单

## 总结

| 维度 | 评分 (1-5) | 说明 |
|-----|-----------|------|
| 可靠性 | ⭐⭐⭐⭐⭐ | 持久化 + 队列 + 重连 |
| 可维护性 | ⭐⭐⭐⭐⭐ | 模块化 + 类型注解 |
| 性能 | ⭐⭐⭐⭐ | 防抖节省 Token，队列有轻微延迟 |
| 功能丰富度 | ⭐⭐⭐⭐⭐ | 完整权限 + 多会话策略 |
| 实施复杂度 | ⭐⭐⭐ | 需要 1-2 周开发时间 |

**总体建议:** 强烈建议实施改进方案，特别是对于生产环境使用的 Bot。
